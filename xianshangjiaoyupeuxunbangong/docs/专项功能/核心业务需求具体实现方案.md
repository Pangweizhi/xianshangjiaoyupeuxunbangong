# 线上教育培训办公系统核心业务需求具体实现方案

## 1. 文档说明

本文依据以下两部分内容整理：

- `docs/专项功能/线上教育培训办公系统——核心业务需求文档.docx`
- 当前仓库中的后端、数据库脚本与前端实现现状

本文目标不是重复需求，而是给出“如何在现有系统上落地”的具体实现方案，重点覆盖数据设计、模块拆分、接口方案、页面方案与实施顺序。

## 2. 现有系统实现基础

当前系统已经具备一部分教学业务基础，可直接复用：

- 课程管理：`kecheng` 表，后端 `KechengController`
- 教学资料管理：`jiaoxueshipin` 表，后端 `JiaoxueshipinController`
- 作业管理：`zuoye` 表，后端 `ZuoyeController`
- 作业提交与批改：`zuoye_submit` 表，后端 `ZuoyeSubmitController`
- 公告管理：`news` 表
- 会议通知：`kaihuitongzhi` 表
- 教师、学生、管理员三类角色与 Token 鉴权
- 新前端已完成课程、资料、作业、公告、会议等基础页面

这说明当前系统已经具备“课程展示、资料发布、作业发布、作业提交”的基础闭环，但距离需求文档中的“课程审核、章节管理、选课学习、考试、学习进度、学分发放”还有明显差距。

## 3. 现状与需求差距

结合需求文档，当前主要缺口如下：

1. 缺少课程审核机制
2. 缺少课程章节模型，当前教学资料未与课程章节形成强绑定
3. 缺少学生选课/我的课程关系表
4. 缺少学习进度跟踪
5. 缺少考试模块
6. 缺少成绩汇总与课程结课判定
7. 缺少学分发放与防重复控制
8. 缺少围绕上述业务的后台菜单、教师端菜单和学生端页面

因此，建议保留现有课程、资料、作业能力，在此基础上新增“审核、章节、选课、进度、考试、学分”六类核心能力。

## 4. 总体实现思路

系统按“教-学-练-考-评-证”主线扩展，业务链路如下：

1. 教师创建课程
2. 管理员审核课程
3. 教师维护课程章节与章节资源
4. 管理员审核章节/资源
5. 学生选课进入我的课程
6. 学生按章节学习视频与课件
7. 系统记录学习进度
8. 教师发布作业与考试
9. 学生提交作业并参加考试
10. 系统汇总作业成绩、考试成绩、学习完成度
11. 满足结课规则后发放学分

## 5. 模块级具体实现方案

## 5.1 课程管理与审核

### 现有基础

- 已有 `kecheng` 表和课程管理页面
- 已有课程名称、封面、类型、时长、教师、详情等字段

### 建议扩展

在 `kecheng` 表新增以下字段：

- `course_status`：课程状态，值建议为 `draft`、`pending_review`、`approved`、`rejected`、`finished`
- `credit_score`：课程学分
- `course_intro`：课程简介，可复用现有详情字段，也可单独拆分
- `review_remark`：审核意见
- `review_time`：审核时间
- `review_admin_id`：审核管理员

### 业务规则

- 教师新建课程后默认状态为 `pending_review`
- 审核通过后课程才允许学生侧展示和选课
- 审核驳回后教师可修改后再次提交
- 已有学生学习记录的课程不允许直接物理删除，只允许停用

### 页面方案

- 管理员端新增“课程审核”页面
- 教师端课程管理页增加“提交审核”“查看审核意见”
- 学生端课程列表仅展示 `approved` 状态课程

## 5.2 课程章节管理

### 新增数据表

建议新增 `course_chapter` 表：

- `id`
- `kecheng_id`
- `chapter_name`
- `chapter_sort`
- `chapter_summary`
- `chapter_status`
- `review_remark`
- `review_time`
- `review_admin_id`
- `create_time`
- `is_deleted`

### 业务规则

- 一个课程对应多个章节
- 章节排序必须唯一
- 章节创建后进入待审核状态
- 只有审核通过章节才能在学生端展示

### 页面方案

- 教师端新增“章节管理”
- 管理员端新增“章节审核”
- 学生端课程详情页新增章节目录

## 5.3 教学资源管理

### 现有基础

当前 `jiaoxueshipin` 更接近“教学资料/备课信息”，但未和课程章节建立明确关系。

### 改造建议

方案建议分两步：

1. 保留原 `jiaoxueshipin` 模块，作为兼容旧数据的资料表
2. 新增 `course_resource` 表，承接章节级资源管理

建议新增 `course_resource` 表字段：

- `id`
- `kecheng_id`
- `chapter_id`
- `resource_name`
- `resource_type`：视频、PPT、文档、压缩包
- `resource_url`
- `cover_url`
- `duration_seconds`
- `resource_status`
- `review_remark`
- `review_time`
- `review_admin_id`
- `create_time`
- `is_deleted`

### 业务规则

- 资源必须挂载到具体章节
- 视频支持时长记录，供学习进度计算
- PPT/文档支持下载
- 管理员可审核资源内容

### 页面方案

- 教师端新增“教学资源管理”
- 资源上传表单中必须选择课程与章节
- 学生端章节详情中区分视频资源与下载资源

## 5.4 学生选课与我的课程

### 新增数据表

新增 `course_enroll` 表：

- `id`
- `kecheng_id`
- `yonghu_id`
- `enroll_status`：已选课、学习中、已结课
- `enroll_time`
- `finish_time`
- `is_deleted`

### 业务规则

- 学生只能选择审核通过的课程
- 同一学生同一课程只能选课一次
- 课程进入“我的课程”后才允许记录学习进度、作业、考试

### 页面方案

- 学生端课程详情增加“立即选课”
- 学生端新增“我的课程”
- 教师端可查看课程选课人数
- 管理员端可查看全局选课情况

## 5.5 学习进度管理

### 新增数据表

新增 `study_progress` 表：

- `id`
- `kecheng_id`
- `chapter_id`
- `resource_id`
- `yonghu_id`
- `study_seconds`
- `progress_percent`
- `is_completed`
- `last_study_time`
- `create_time`

### 进度计算规则

- 视频资源按播放时长累计
- 当播放进度达到 90% 以上时，该资源记为已完成
- 章节下全部资源完成，则章节完成
- 课程下全部必学章节完成率达到设定阈值后，课程学习条件达标

### 页面方案

- 学生端“我的课程”展示课程总进度
- 课程学习页展示章节完成状态
- 教师端新增“学生学习进度”页面

## 5.6 作业模块优化

### 现有基础

- 已有 `zuoye` 表
- 已有 `zuoye_submit` 表
- 学生端已支持提交附件，教师端已可做批改

### 建议扩展

在 `zuoye` 表新增：

- `kecheng_id`
- `chapter_id`
- `deadline_time`
- `score_total`
- `publish_status`

在 `zuoye_submit` 表补充或规范：

- `submit_score`
- `submit_status`
- `check_time`
- `submit_remark`

### 业务规则

- 作业必须归属到课程，可选关联章节
- 超过截止时间默认禁止提交，管理员可配置是否允许补交
- 作业成绩达到 60 分视为通过

## 5.7 考试模块

### 新增数据表

建议新增以下表：

- `exam`：考试基本信息
- `exam_question`：题库
- `exam_paper_question`：试卷与题目关系
- `exam_record`：学生考试记录
- `exam_answer`：学生答题明细

`exam` 表建议字段：

- `id`
- `kecheng_id`
- `exam_name`
- `exam_type`
- `total_score`
- `pass_score`
- `duration_minutes`
- `start_time`
- `end_time`
- `paper_mode`：固定组卷、随机组卷
- `publish_status`
- `create_time`

`exam_question` 表建议字段：

- `id`
- `kecheng_id`
- `question_type`：单选、多选、判断、简答
- `question_title`
- `option_a` 至 `option_d`
- `right_answer`
- `question_score`
- `analysis`

`exam_record` 表建议字段：

- `id`
- `exam_id`
- `yonghu_id`
- `total_score`
- `pass_flag`
- `start_time`
- `submit_time`
- `check_status`

### 评分规则

- 单选、多选、判断题自动判分
- 简答题由教师人工批改
- 全部题目完成判分后生成最终考试成绩
- 考试成绩达到 60 分或达到考试设定及格线视为通过

### 页面方案

- 管理员端新增“考试管理”
- 教师端新增“考试管理”“题库管理”“阅卷管理”
- 学生端新增“考试信息”“在线考试”“我的成绩”

## 5.8 学分发放与结课判定

### 新增数据表

新增 `course_credit_record` 表：

- `id`
- `kecheng_id`
- `yonghu_id`
- `credit_score`
- `grant_status`
- `grant_time`
- `grant_remark`
- `source_rule_snapshot`

### 结课规则建议

默认规则可配置为：

- 课程学习进度达到 100% 或不低于 90%
- 作业成绩大于等于 60 分
- 考试成绩大于等于 60 分

满足条件后：

- 系统自动发放课程学分
- 同一课程同一学生只允许发放一次
- 若课程已发放学分，再次重算时只能更新状态，不得重复插入记录

### 实现建议

- 在成绩变更、考试批改完成、学习进度更新后触发结课判定
- 结课判定逻辑集中放在独立服务中，避免散落在多个 Controller

## 6. 后端实现方案

## 6.1 代码层次建议

延续当前 Spring Boot + MyBatis-Plus 结构，新增以下模块：

- `CourseChapterController/Service/Dao/Entity`
- `CourseResourceController/Service/Dao/Entity`
- `CourseEnrollController/Service/Dao/Entity`
- `StudyProgressController/Service/Dao/Entity`
- `ExamController/Service/Dao/Entity`
- `ExamQuestionController/Service/Dao/Entity`
- `ExamRecordController/Service/Dao/Entity`
- `CourseCreditRecordController/Service/Dao/Entity`
- `CourseReviewController` 或在现有课程控制器中补充审核接口

## 6.2 关键接口建议

建议新增以下典型接口：

- `POST /kecheng/submitReview`
- `POST /kecheng/review`
- `GET /courseChapter/page`
- `POST /courseChapter/save`
- `POST /courseChapter/review`
- `POST /courseResource/save`
- `POST /courseEnroll/save`
- `GET /courseEnroll/myCourses`
- `POST /studyProgress/saveOrUpdate`
- `POST /exam/save`
- `POST /exam/publish`
- `POST /examRecord/start`
- `POST /examRecord/submit`
- `POST /examRecord/check`
- `GET /score/myPage`
- `GET /credit/myPage`

## 6.3 业务服务建议

建议抽出三个核心服务：

- `CourseReviewService`：处理课程、章节、资源审核
- `LearningProgressService`：处理学习进度累计与课程完成度计算
- `CourseSettlementService`：处理作业成绩、考试成绩、学分发放与结课状态

这样可以避免把复杂逻辑直接写进现有 Controller。

## 8. 数据迁移与兼容方案

考虑当前系统已有旧数据，建议采用兼容式升级：

1. 保留现有 `kecheng`、`jiaoxueshipin`、`zuoye`、`zuoye_submit`
2. 通过新增字段和新增表完成扩展
3. 原 `jiaoxueshipin` 数据可逐步迁移到 `course_resource`
4. 若短期不迁移旧数据，可在前端用“旧资料”“章节资源”并存方式平滑过渡

## 9. 推荐实施顺序

建议按四期实施，降低改造风险：

### 第一期：课程审核与章节管理

- 扩展课程状态字段
- 新增课程审核
- 新增章节表与章节管理

### 第二期：资源管理与选课学习

- 新增章节资源表
- 新增选课关系表
- 新增我的课程与学习进度

### 第三期：作业升级与考试模块

- 作业关联课程/章节
- 新增考试、题库、考试记录、阅卷功能

### 第四期：成绩汇总与学分发放

- 汇总学习、作业、考试成绩
- 结课判定
- 学分自动发放
- 统计报表

## 10. 预期成果

方案落地后，系统将从当前“课程+资料+作业”的基础教学系统，升级为完整的：

- 课程审核体系
- 章节化学习体系
- 学生选课与进度跟踪体系
- 作业与考试评价体系
- 结课与学分发放体系

最终形成需求文档要求的“教-学-练-考-评-证”一体化核心业务闭环。
