# 智能问答助手具体实现方案

## 1. 文档说明

本文说明智能问答助手在当前系统中的实现方式、当前边界和后续扩展方向。

截至 `2026-05-05`，该专项的一期能力已经实际落地，本文更偏向“现状说明 + 当前方案沉淀”。

## 2. 建设目标

智能问答助手要解决的不是通用聊天，而是系统内的教学与操作辅助问题。

核心目标包括：

- 提供统一“问 AI”入口
- 支持多轮会话与历史会话
- 会话数据按用户隔离并持久化
- 结合系统业务数据做回答
- 对系统无关问题进行友好收口

## 3. 当前可复用基础

当前系统已经具备以下可直接复用能力：

- 用户身份与角色体系
- Token 鉴权
- 课程、章节、资源数据
- 作业、考试、成绩数据
- 新前端学生端与后台端页面框架

因此，智能问答助手不需要从零建设用户体系或业务模型，只需要补：

- AI 会话与消息存储
- AI 问答服务
- 业务上下文拼装
- 问 AI 页面与入口

## 4. 当前总体方案

整体采用：

“前端对话页 + 后端 AI 服务 + 系统业务上下文 + MySQL 持久化 + 模型网关/规则回退”

主流程如下：

1. 前端进入问 AI 页面
2. 前端传入当前用户、当前页面或当前课程等上下文
3. 后端构造提示词与业务上下文
4. 优先走模型网关
5. 如模型不可用或命中规则问答场景，则走规则回答
6. 把会话与消息写入数据库
7. 前端展示历史会话与多轮消息

### 7.2 暂未承诺

- 完整知识库问答

- PPT 原文级精准问答
- 教材级 RAG 检索增强

## 8. 为什么当前先不做完整 RAG

原因很直接：

1. 当前系统已经有大量结构化业务数据，可以先做出可用的一期成果
2. 直接上 RAG 会引入文本抽取、切片、向量库、召回评估等额外复杂度
3. 现阶段更需要可落地、可展示、可维护的系统内助手

因此当前方案优先保证：

- 能问
- 能记住历史
- 能按用户隔离
- 能回答系统内常见问题

## 9. 与核心业务的联动关系

智能问答助手与核心业务并不是分离的，它本质上是核心业务的辅助层。

当前已经能围绕以下业务提供辅助：

- 课程学习
- 学习进度
- 作业
- 考试
- 成绩
- 菜单导航

## 10. 后续扩展方向

后续如果继续推进，建议按这个顺序扩展：

1. 浏览器端真实问答体验调优
2. 提示词模板细化
3. 章节资源文本抽取
4. 轻量 RAG
5. 教师自动出题
6. 更复杂的函数调用与工具编排
