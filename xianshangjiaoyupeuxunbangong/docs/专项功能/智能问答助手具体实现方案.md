# 智能问答助手具体实现方案

## 1. 文档说明

本文依据以下内容整理：

- `docs/专项功能/智能问答助手需求文档.docx`
- 当前仓库已有的 Spring Boot 后端、MySQL 数据库、新前端 `admin-web/front-web/shared`
- 当前系统已经具备的课程、章节、资源、作业、考试、公告、会话鉴权等基础能力

本文目标不是重复需求，而是明确“如何在现有系统上落地智能问答助手”，并给出首期边界、技术方案、数据结构、接口设计、前端交互和后续扩展路径。

截至 `2026-05-03`，本专项建议按“两阶段”推进：

1. 首期先落地通用问答、系统导航问答、会话历史、用户隔离、提示词约束、系统数据 Function Call 接入。
2. 二期再按课程学习场景补齐 RAG/章节内容摘要/教师自动出题等增强能力。

## 2. 需求目标与首期边界

结合需求文档，智能问答助手的核心目标包括：

- 在系统内提供“问AI”统一入口
- 支持类似“豆包”的多轮会话与历史会话
- 会话数据按用户隔离，并持久化到 MySQL
- 当用户问题与系统业务相关时，可结合系统数据回答
- 当用户问题与系统无关时，进行友好拒答
- 在学生端提供学习辅助与系统导航能力
- 在教师端提供教学辅助与后续自动出题扩展基础

首期边界建议如下：

- 先不做完整知识库后台，不引入独立向量库
- 先使用 Spring AI + 阿里百炼大模型 + MySQL 持久化实现
- 先支持“纯对话 + 业务 Function Call + 猜你想问”
- 对课程视频、PPT 的“基于原文精确问答”先不承诺完全可用
- 但为后续 RAG 预留表结构、服务接口和资源切片能力

这样做的原因是：当前系统已经有课程、章节、资源、作业、考试等业务数据，先做“系统内智能助手”可以最快形成可演示成果，且不会过早把复杂度压到知识库建设和文本抽取上。

## 3. 现有系统可复用基础

当前系统已经具备以下可直接复用能力：

- 用户体系与角色区分：管理员、教师、学生
- Token 鉴权与用户身份识别
- 课程数据：`kecheng`
- 章节数据：`course_chapter`
- 资源数据：`course_resource` 与旧表 `jiaoxueshipin`
- 作业数据：`zuoye`、`zuoye_submit`
- 考试数据：`exam`、`exam_question`、`exam_record`
- 公告与通知：`news`、`kaihuitongzhi`
- 新前端已有后台与学生端页面框架

因此，智能问答助手不需要从零构建用户、课程或资源模型，而是在现有系统上新增：

- AI 会话与消息存储
- AI 问答服务
- 系统业务查询工具函数
- 问AI入口与对话页面

## 4. 总体实现思路

整体采用“前端问答页 + Spring AI 编排层 + 业务工具函数 + MySQL 持久化”的方案：

1. 前端点击“问AI”按钮进入对话页。
2. 前端传入当前用户、当前页面场景、当前课程/章节等上下文。
3. 后端根据用户身份、页面场景、提示词模板构造对话请求。
4. 大模型优先基于系统提示词回答。
5. 如判断问题与系统业务相关，则通过 Function Call 查询课程、作业、考试、资源等系统数据，再组织给模型生成最终答案。
6. 对话结果和消息摘要写入 MySQL，支持历史会话和分页查看。
7. 对无关问题执行拒答策略，避免模型跑题。

首期不做完整 RAG 时，学习辅助能力主要依靠三类信息：

- 当前页面上下文
- 当前课程/章节/资源元数据
- 系统中可直接读取的业务数据

二期如要增强“基于视频/PPT内容回答”，再补：

- 资源文本抽取
- 章节摘要生成
- 向量检索或基于 MySQL 的轻量切片召回

## 5. 技术选型

建议技术栈如下：

- 大模型接入：阿里百炼 OpenAI 兼容接口
- 模型平台：阿里百炼
- 数据库存储：MySQL
- 缓存与限流：首期可先用本地限流，后续可加 Redis
- 前端：现有 Vue 3 + TypeScript 前后台工程

原因：

- 当前项目为 `Spring Boot 2.2 + Java 8`，直接接入新版本 Spring AI 兼容成本较高
- 阿里百炼提供 OpenAI 兼容接口，适合在现有项目中以最小代价接入
- MySQL 可满足会话持久化与用户隔离
- 当前系统已有统一后端，便于把业务查询工具直接挂进 AI 服务

补充说明：

- 如果后续项目整体升级到 `Spring Boot 3.x`，再切换到 Spring AI 会更合适
- 当前二期实现以“兼容接口直连 + 可回退规则应答”为主

## 6. 分角色应用场景

## 6.1 学生端

首期重点支持两类能力：

1. 学习辅助问答
- 当前课程是什么
- 当前章节讲什么
- 帮我总结这一节
- 这个概念是什么意思
- 给我举个例子

2. 系统导航问答
- 作业在哪里看
- 课程资料在哪里下载
- 考试入口在哪
- 我的成绩怎么看

其中“学习辅助问答”首期以章节标题、资源标题、课程简介、教师已发布内容为基础；二期再增强到资源正文级别理解。

## 6.2 教师端

首期重点支持：

- 系统导航问答
- 教学管理问答
- 课程、作业、考试状态查询

二期增强：

- 自动出题
- 按章节生成题目
- 生成试卷与解析

## 6.3 管理员端

首期可支持：

- 课程审核规则说明

- 核心报表口径解释

## 7. 系统架构设计

建议新增以下后端模块：

- `AiChatController`
- `AiChatSessionService`
- `AiChatMessageService`
- `AiAssistantService`
- `AiPromptService`
- `AiToolService`

职责划分建议：

- `AiChatController`
  - 提供创建会话、会话列表、消息列表、发送消息、删除会话等接口
- `AiAssistantService`
  - 统一编排大模型调用、工具调用、拒答策略、答案落库
- `AiPromptService`
  - 维护系统提示词、角色提示词、场景提示词
- `AiToolService`
  - 对接课程、章节、资源、作业、考试、成绩等查询函数
- `AiChatSessionService/AiChatMessageService`
  - 负责会话和消息的 CRUD、轮次限制、归档清理

## 8. 数据库设计

首期建议新增 3 张表。

## 8.1 `ai_chat_session`

用于存储会话头信息。

建议字段：

- `id`
- `user_id`
- `user_table`
- `role_type`
- `session_title`
- `biz_scene`
- `course_id`
- `chapter_id`
- `resource_id`
- `last_message_at`
- `message_count`
- `status`
- `insert_time`
- `update_time`

说明：

- `user_table` 兼容当前系统多角色表结构
- `biz_scene` 用于区分学生学习场景、系统导航场景、教师出题场景
- `course_id/chapter_id/resource_id` 用于绑定当前上下文

## 8.2 `ai_chat_message`

用于存储会话消息。

建议字段：

- `id`
- `session_id`
- `user_id`
- `role_type`
- `message_role`
- `message_type`
- `content`
- `tool_name`
- `tool_request_json`
- `tool_response_json`
- `token_estimate`
- `sort_no`
- `insert_time`

说明：

- `message_role` 取值建议为 `system/user/assistant/tool`
- `message_type` 可区分普通文本、猜你想问、工具结果、拒答结果
- `tool_request_json/tool_response_json` 用于审计 Function Call 过程

## 8.3 `ai_chat_context_snapshot`

用于存储发送当下的上下文快照，避免后续课程数据变化导致会话语义不一致。

建议字段：

- `id`
- `session_id`
- `message_id`
- `scene_code`
- `context_json`
- `insert_time`

如果首期希望控制复杂度，也可以先不单独建表，而是把上下文快照直接存到 `ai_chat_message.context_json` 中。

## 9. 会话持久化与存储控制

需求文档明确要求避免 MySQL 被会话数据打爆，因此首期必须带上存储控制策略。

建议规则：

1. 每个用户最多保留 `50` 个会话。
2. 每个会话最多保留 `100` 条消息。
3. 超过上限后，优先删除最旧会话或最旧消息。
4. 对超长内容只保留摘要，不保存完整大模型中间推理信息。
5. 工具调用 JSON 做长度截断，避免单条消息过大。
6. 定时任务清理长时间未访问且状态为关闭的旧会话。

附加建议：

- 会话列表只返回最近消息摘要
- 消息详情分页加载，不一次性返回全部
- 后续如会话量上升，再补分表或归档表

## 10. 提示词与拒答策略

智能问答助手必须被约束在“系统相关问答”范围内。

建议采用三层提示词：

1. 平台级系统提示词
- 定义身份：你是线上教育培训办公系统的智能问答助手
- 定义边界：只回答与当前系统、课程学习、教学管理相关的问题
- 定义拒答策略：与系统无关的问题需礼貌拒答，并引导用户回到系统场景

2. 角色提示词
- 学生侧偏学习辅助与操作引导
- 教师侧偏课程管理、作业考试辅助
- 管理员侧偏审核、统计、流程说明

3. 页面场景提示词
- 课程详情页
- 作业页
- 考试页
- 后台课程管理页

建议拒答口径：

- 若问题与系统无关，不直接闲聊展开
- 用一段简短话术说明当前助手聚焦课程学习与系统使用
- 再给出 2 个可继续提问的推荐问题

这部分可与 `docs/专项功能/调教AI的提示词.md` 联动维护。

## 11. Function Call 设计

首期的核心价值，不在于“纯聊天”，而在于“能读系统数据”。

建议首期接入以下工具函数：

1. `getCurrentUserProfile`
- 获取当前用户角色、姓名、所属班级/课程范围

2. `getCourseSummary`
- 根据 `courseId` 返回课程简介、教师、章节数、资源数、学分、审核状态

3. `getChapterSummary`
- 根据 `chapterId` 返回章节标题、排序、关联资源列表

4. `getMyCourseProgress`
- 返回学生当前课程学习进度、作业完成情况、考试通过情况、是否结课

5. `getHomeworkListForUser`
- 返回当前学生或教师可见的作业列表

6. `getExamListForUser`
- 返回当前学生或教师可见的考试列表

7. `getSystemNavigationTips`
- 根据页面标识返回操作路径说明，例如“查看作业在哪里”

8. `getTeacherQuestionGenerationContext`
- 二期出题时返回课程、章节、已有题目样例、题型约束

Function Call 使用原则：

- 只暴露只读查询函数
- 每个函数强制带当前登录用户上下文
- 服务端二次校验权限，避免越权读取
- 模型不能直接拼 SQL，只能调用受控工具函数

## 12. 首期是否做 RAG

结论：首期不做完整 RAG，但保留可扩展设计。

原因：

- 当前需求最迫切的是“系统内可用的会话助手”
- 课程视频、PPT 的正文抽取、清洗、切片、召回、评估成本较高
- 现有系统资源结构尚在演进，先做稳定入口和业务工具更稳

但以下情况满足时，可以直接进入二期 RAG：

- 系统已有可下载的 PPT/文档资源
- 可以抽取文本内容
- 学习辅助问答必须回答到章节正文级别

二期 RAG 方案建议：

1. 新增资源文本抽取任务
2. 生成章节摘要与资源切片
3. 建立 `ai_knowledge_segment` 一类轻量表
4. 先使用 MySQL 存切片与关键词召回
5. 若效果不足，再接入向量库

这样能避免一开始就把系统复杂度拉满。

## 13. 后端接口设计

建议新增以下接口。

## 13.1 会话接口

- `GET /aiChat/session/page`
  - 分页查询当前用户会话
- `POST /aiChat/session/create`
  - 创建新会话
- `POST /aiChat/session/rename`
  - 修改会话标题
- `POST /aiChat/session/delete`
  - 逻辑删除会话

## 13.2 消息接口

- `GET /aiChat/message/page`
  - 分页查询某会话消息
- `POST /aiChat/message/send`
  - 发送消息并返回 AI 回复

`send` 请求建议包含：

- `sessionId`
- `content`
- `bizScene`
- `courseId`
- `chapterId`
- `resourceId`
- `pageCode`

## 13.3 猜你想问接口

- `GET /aiChat/recommendQuestions`

返回 2 个推荐问题，按场景动态生成：

- 学生课程详情页：总结本章、这个章节重点是什么
- 学生考试页：考试入口在哪、我还能参加几次考试
- 教师课程页：如何新增章节、如何给本课程发布作业

## 14. 前端交互方案

## 14.1 问AI按钮

需求文档要求该按钮：

- 颜色醒目但不突兀
- 有动态高亮效果
- 参考淘宝 APP “问AI”按钮
- 文字旁带“见图知意”的图标感觉

具体建议：

- 学生端放在顶部导航右侧或页面右下悬浮入口
- 后台教师端放在头部工具栏右侧
- 按钮使用渐变底色、轻微呼吸动画、圆角胶囊形态
- 图标使用“对话气泡 + 星光/灵感”组合，而不是夸张机器人图标

## 14.2 问答页布局

建议采用三栏或二栏简化结构：

- 左侧：历史会话列表
- 右侧主区：消息流 + 输入框
- 顶部：当前场景标识，例如“课程学习助手”“系统导航助手”

核心交互：

- 新建会话
- 会话标题自动生成
- 消息流展示用户与 AI 回复
- 显示猜你想问
- 支持继续追问
- 支持从课程页带上下文进入

## 14.3 猜你想问

每次进入对话页时，根据角色和场景返回 2 个问题：

- 学生课程页
  - “帮我总结这一节内容”
  - “这个章节的重点是什么”

- 学生系统导航页
  - “作业在哪里查看”
  - “考试入口在哪”

- 教师课程页
  - “如何给本课程新增作业”
  - “帮我梳理本课程当前章节结构”

## 15. 权限与安全控制

需要重点控制以下风险：

1. 用户数据隔离
- 会话只允许本人查看
- 工具函数只查询当前用户有权访问的数据

2. 提示词注入
- 用户输入不允许覆盖系统提示词
- 工具调用必须由后端白名单控制

3. 越权问答
- 学生不能通过提问拿到其他学生成绩
- 教师不能读取非本人课程敏感数据，除非其角色本身有权限

4. 成本控制
- 限制单次提问长度
- 限制每分钟请求次数
- 限制每日会话调用总量

5. 审计能力
- 保留会话、消息、工具调用简要日志
- 便于后续排查错误回复与越权风险

## 18. 结论

智能问答助手专项最合适的落地路径，不是一步到位做成完整知识库产品，而是先基于现有系统数据和 Spring AI 做出“系统内可用的智能助手”。

首期重点应放在：

- 问AI统一入口
- 多轮会话与历史会话
- MySQL 持久化与存储控制
- 用户隔离
- 提示词约束与友好拒答
- 系统数据 Function Call
- 猜你想问
