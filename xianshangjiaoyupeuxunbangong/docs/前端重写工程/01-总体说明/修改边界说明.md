# 修改边界说明

## 1. 目的

为了避免后续修改再次落回旧页面或旧资源目录，统一约定当前仓库的前端修改边界。

## 2. 前端修改边界

### 2.1 旧学生端目录

以下目录默认视为只读，不再继续做页面和样式改动：

- `src/main/resources/front/front`

### 2.2 旧后台目录

以下目录默认视为只读，不再继续做页面和样式改动：

- `src/main/resources/admin/admin`

### 2.3 新学生端目录

所有“前台 / 学生端”相关的视觉、交互、布局、页面逻辑改动，只允许放到：

- `new-frontend/front-web`

### 2.4 新管理端目录

所有“后台 / 教师端 / 管理端”相关的视觉、交互、布局、页面逻辑改动，只允许放到：

- `new-frontend/admin-web`

### 2.5 共享层

如果需要同步改动以下内容，统一放到共享层或后端接口层：

- 共享类型
- 共享请求封装
- 登录态处理
- 通用工具

对应目录：

- `new-frontend/shared`

## 3. 允许修改的后端范围

当页面改动依赖后端支持时，可以修改：

- 控制器
- 服务层
- DAO / Mapper
- 实体
- 数据库脚本

但原则是：

- 优先补当前主链路
- 不随意推翻现有接口风格
- 数据结构变更要同步文档

## 4. 默认解释规则

后续如果需求只说“前端改一下”或“后台改一下”，默认按下面理解：

- “前端” 指 `new-frontend/front-web`
- “后台” 指 `new-frontend/admin-web`
- “共享逻辑” 指 `new-frontend/shared`

默认不再回写旧前端和旧后台资源目录。

## 5. 例外情况

只有在以下情况，才考虑动旧目录：

1. 用户明确要求修改旧前端
2. 新前端当前确实没有覆盖对应能力，且短期无法补齐
3. 必须为了兼容线上旧入口做最小修复

如果属于例外情况，应该在提交前明确说明原因。

## 6. 当前阶段说明

截至 `2026-05-05`，该边界规则优先级很高。

原因很明确：

- 当前新前端已经覆盖核心业务主链路
- 继续同时改新旧两套前端会迅速失控
- 文档、脚本、页面已经开始围绕 `new-frontend` 持续同步
