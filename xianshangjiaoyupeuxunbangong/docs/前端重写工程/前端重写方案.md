# 前端重写方案

## 1. 目标

当前项目后端接口和数据库已经可用，旧前端只做保留，不做大改动。新方案的目标是：

- 新做一套**前台视觉系统**
- 新做一套**后台管理系统**
- **复用现有后端接口**
- 逐步替换旧前端，不影响后端稳定运行

核心原则：

- 后端不动，前端重写
- 先统一体验，再逐步补功能
- 新旧前端可以并行存在

## 2. 现状判断

### 2.1 旧前台

旧前台位于 `src/main/resources/front/front`，属于静态页面资源，混用了多种技术风格：

- HTML/CSS/JS
- Vue.js
- layui
- jQuery
- Bootstrap

它适合“保持可运行”，不适合继续大规模视觉改造。

### 2.2 旧后台

旧后台位于 `src/main/resources/admin/admin`，是 Vue2 + Element UI 的生成式管理端，模块已经齐全，但视觉和交互都偏旧。

### 2.3 后端接口

后端是 Spring Boot + MyBatis-Plus，接口已经按角色和业务模块拆好，适合继续复用。

## 3. 总体方案

建议将前端拆成两个新的独立工程：

- `front-web`：新前台视觉系统
- `admin-web`：新后台管理系统

这两个前端都直接调用现有后端接口，不改业务逻辑，只改展示层。

### 推荐结构

```text
new-frontend/
  front-web/
  admin-web/
  shared/
```

- `front-web` 负责学生、教师可见的展示和交互
- `admin-web` 负责管理员运营、配置、数据管理
- `shared` 负责可复用的 API 封装、权限工具、请求拦截器、通用组件、样式变量

## 4. 技术选型建议

### 4.1 前台视觉系统

推荐：

- `Vue 3`
- `Vite`
- `Pinia`
- `Vue Router`
- `Axios`
- `Tailwind CSS` 或 `SCSS + 自定义设计系统`

原因：

- 适合做更现代、更统一的视觉风格
- 组件拆分和样式维护成本更低
- 更适合后续持续扩展

如果团队更偏稳妥，也可以继续用 `Vue 2`，但从“重写一套前端”的角度看，前台建议直接上 `Vue 3`。

### 4.2 后台管理系统

推荐：

- `Vue 3`
- `Vite`
- `Element Plus`
- `Pinia`
- `Vue Router`
- `Axios`

如果短期内追求迁移成本低，也可以先用 `Vue 2 + Element UI` 重构样式，但长期更建议直接升级到 `Vue 3 + Element Plus`。

## 5. 接口复用方式

新前端直接复用现有后端接口，重点处理以下几类内容：

### 5.1 登录与身份

当前后端通过 `Token` 请求头识别身份，登录后会返回：

- `token`
- `role`
- `userId`
- `tableName`

新前端需要统一做：

- 登录接口封装
- token 持久化
- 请求拦截器自动带上 `Token`
- 失效后跳回登录页

### 5.2 角色控制

后端已经按角色做了数据过滤，新前端只需要在前端层面控制导航显示和页面入口：

- 管理员看到管理菜单
- 教师看到教学业务菜单
- 学生看到学习和互动菜单

真正的数据权限仍然由后端控制，前端只负责体验层。

### 5.3 公共接口封装

建议把接口按模块拆开：

- `auth`
- `user`
- `teacher`
- `student`
- `course`
- `homework`
- `forum`
- `notice`
- `dictionary`
- `config`
- `file`

这样可以避免以后接口越来越乱。

- 卡片化布局
- 更明确的色彩主题
- 更少表格感，更多内容展示感
- 适配手机端和桌面端

### 6.2 新后台管理系统

建议优先做这些模块：

- 登录页
- 首页仪表盘

- 教师管理

- 列表页统一搜索、分页、表单弹窗
- 表格不要堆太多无层次元素

## 7. 样式重写策略

### 7.1 不建议直接在旧代码上硬改

旧代码最大的问题不是“颜色不好看”，而是：

- 页面结构杂
- 样式来源多
- 组件风格不统一
- 维护边界差

所以重写时建议直接建立新的设计系统，而不是逐个旧页面去补丁式修样式。

### 7.2 先统一设计变量

建议先定义：

- 主色
- 辅助色
- 成功/警告/错误色
- 字体大小体系
- 间距体系
- 圆角体系
- 阴影体系

后续所有页面都从这些变量生成，不要每个页面单独写一套风格。

### 7.3 组件优先

建议先沉淀基础组件：

- `AppLayout`
- `PageHeader`
- `SearchBar`
- `StatCard`
- `DataTable`
- `FormDialog`
- `EmptyState`
- `UploadField`
- `RichTextEditor`
- `ConfirmButton`

先做组件，再做页面，避免重复造轮子。

- 路由守卫控制未登录访问
- 菜单权限控制不同角色显示不同模块

真正的接口权限仍然依赖后端拦截器和业务判断。

## 9. 迁移步骤

### 第 1 阶段：搭框架

- 建立 `front-web` 和 `admin-web`
- 搭建路由、请求层、布局层、通用组件
- 接好登录和 token 机制

### 第 2 阶段：做视觉骨架

- 首页
- 登录页
- 基础列表页
- 基础详情页
- 基础表单页

### 第 3 阶段：迁移核心业务

- 课程
- 公告
- 作业
- 论坛
- 备课
- 会议通知
- 用户中心
- 管理后台核心模块

### 第 4 阶段：优化体验

- 响应式布局
- 加载态
- 空状态
- 搜索体验
- 表单校验
- 统一弹窗与通知

### 第 5 阶段：替换旧前端入口

- 旧前端保留为兜底
- 新前端作为主入口
- 逐步把访问入口切换到新项目

## 10. 风险与注意事项

- 旧前端和新前端并存时，要避免接口变更造成互相影响
- 由于后端当前很多接口是按角色隐式过滤，前端必须严格处理登录态
- 旧代码里有一些命名不规范的模块，例如 `dianfnag`，新前端建议统一命名映射，但不要直接改后端表名，避免引入连锁问题
- 图片、文件上传、富文本编辑器这类能力要尽早统一封装，否则后面改页面会很痛苦

## 11. 建议结论

如果你的目标是“把现在这套前端彻底变好看，并且后续能持续迭代”，最佳方案是：

- **后端保留**
- **前台重写**
- **后台重写**
- **统一调用现有接口**
- **先搭设计系统，再做页面**

这比在旧前端上反复修样式更稳，也更省后期维护成本。

### 14.2 接口封装规范

#### 统一请求层

建议单独封装 `request` 实例：

- 统一 `baseURL`
- 统一超时
- 统一请求头
- 统一 token 注入
- 统一错误提示
- 统一登录失效处理

#### 接口模块化

每个业务模块只放自己相关的接口，不把所有接口塞进一个文件。

例如：

- `api/course.ts` 只处理课程
- `api/forum.ts` 只处理论坛
- `api/auth.ts` 只处理登录、注册、退出

#### 返回值规范

建议前端对后端返回统一做一层适配：

- 成功只取 `data`
- 错误统一抛出
- 列表页统一识别分页结构

#### 命名规范

- 页面组件使用 `PascalCase`
- 工具函数使用 `camelCase`
- 接口函数以动词开头，例如 `getCoursePage`、`createForumPost`
- 类型定义和接口名称保持一致，避免后期混乱

### 14.3 权限与状态管理规范

建议状态管理里至少包含：

- `token`
- `role`
- `userInfo`
- `menuList`
- `permissionList`

路由守卫需要完成：

- 未登录跳转登录页
- 不同角色进入不同首页
- 登录后禁止访问登录页

### 14.4 样式与主题规范

为了保证现代化和高可扩展性，建议全局统一：

- 颜色变量
- 字体变量
- 间距变量
- 圆角变量
- 阴影变量
- 断点变量

页面只使用这些变量，不要直接散写魔法数字。

### 14.5 组件化规范

建议优先沉淀这些公共组件：

- `AppHeader`
- `AppSidebar`
- `PageCard`
- `PageSection`
- `DataTable`
- `SearchForm`
- `FormDialog`
- `DetailDrawer`
- `UploadField`
- `ImagePreview`
- `RichTextViewer`
- `RichTextEditor`

组件只负责展示和交互，不直接写业务接口。

### 14.6 可扩展性落地原则

要做到“效果好、现代化、可扩展性高”，核心不是单纯换框架，而是以下几点同时成立：

- 设计系统统一
- 布局统一
- 组件统一
- 接口统一
- 状态统一
- 权限统一

只要这六项做到了，新前端后续加模块不会越做越乱。
